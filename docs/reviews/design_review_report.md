# 設計書総合レビュー報告書

## 1. レビュー対象
- `PROJECT_SPEC.md`

- `docs/design/semantic-mapping-screen.md`
- `docs/design/analysis-run-screen.md`
- `docs/design/results-visualization-screen.md`
- `docs/design/segment-creation-screen.md`

## 2. 総評
主要5画面の導線、画面内状態、GraphQL入出力案、初期アルゴリズム方針は一通り定義されており、プロジェクトのコアバリューである「ビジネスユーザー向けのノーコード分析」には概ね沿っている。特に、各画面で技術用語を業務文脈に置き換える方針が明記されている点は良い。

一方で、現状の設計書は画面単位の粒度が中心で、横断的な設計が不足している。とくに、保存先、GraphQLスキーマの共通型、非同期ジョブ制御、権限・監査、PII/個人情報の扱い、Fabric制約への対応が未確定のまま残っている。実装前にこれらを共通設計として補う必要がある。

## 3. 優先度付き指摘

### P0: 実装前に解消すべき事項

#### 3.1 マッピング・分析・セグメント定義の保存先が未確定
セマンティック・マッピングでは「Fabric側に置くか、アプリ専用ストアに置くか」が未決定であり、分析実行、結果保存、セグメント作成でも同様に永続化責務が曖昧である。

影響:
- 下書き保存、バージョン管理、再実行、監査ログ、権限判定の実装方針が決まらない。
- GraphQL APIがFabricのデータ取得専用なのか、アプリ状態管理も担うのかが不明確になる。

改善提案:
- `docs/design/data-persistence.md` を追加し、以下を明記する。
  - アプリ専用メタデータストアの有無
  - `SemanticMappingDocument`、`AnalysisRunDocument`、`AnalysisResultDocument`、`SegmentDraft` の保存先
  - バージョン番号、楽観ロック、更新者、更新日時、削除/アーカイブ方針
  - Fabricデータ本体とアプリメタデータの責務分界

#### 3.2 GraphQLの共通スキーマ仕様が不足
各画面にGraphQL例はあるが、共通ID、エラー形式、ページング、認可エラー、部分結果、JSON scalarの扱いが統一されていない。

影響:
- 画面ごとにAPI実装の解釈が分かれ、型安全性とエラーハンドリングが崩れやすい。
- Fabric由来の制約や遅延を画面側で一貫して扱えない。

改善提案:
- `docs/design/graphql-contract.md` を追加する。
- 共通型として `ApiError`、`PageInfo`、`AccessDeniedReason`、`JobProgress`、`ValidationIssue` を定義する。
- `ruleTree`、`config`、`outputConfig` を JSON scalar で扱うか、明示input型に分解するかを決定する。
- 一覧取得にはページング、ソート、フィルタ引数を定義する。

#### 3.3 権限・認可・監査ログの仕様が不足
データセット選択では権限不足表示があるが、以降の画面で「誰が何を見られるか」「誰がセグメントを書き込めるか」が未定義である。

影響:
- Fabric上のワークスペース権限とアプリ内操作権限の整合が取れない。
- セグメント作成で顧客テーブルへフラグ付与する場合、誤更新や不正出力のリスクが高い。

改善提案:
- `docs/design/security-and-permissions.md` を追加する。
- Azure Identity/Fabric認可を前提に、操作ごとの必要権限を定義する。
  - データセット閲覧
  - サンプル値閲覧
  - マッピング保存
  - 分析ジョブ開始
  - 結果閲覧
  - CSV出力
  - 顧客フラグ付与
  - 外部連携
- 監査ログ対象として、データセット選択、分析開始、結果保存、CSV出力、セグメント実行を明記する。

#### 3.4 PII/個人情報とサンプル値の扱いが未定義
セマンティック・マッピング、結果可視化、セグメント作成ではサンプル値、顧客ID、氏名または匿名ID、属性値を表示する可能性があるが、マスキングや出力制限が定義されていない。

影響:
- ビジネスユーザー向け画面で不要な個人情報を露出する可能性がある。
- CSV出力や外部連携でデータ持ち出し制御が弱くなる。

改善提案:
- サンプル値はデフォルトでマスクし、権限がある場合のみ表示する。
- `SegmentPreviewRow.displayName` は原則匿名化し、表示可能条件を明記する。
- CSV出力時の列選択、最大件数、ダウンロード監査、期限付きURLの有無を定義する。
- 外部連携は初期版では非表示または管理者設定必須にする。

### P1: 設計品質を上げるために補うべき事項

#### 3.5 画面遷移間の受け渡しモデルが分断されている
`SelectedDatasetContext`、`SemanticMappingDocument`、`AnalysisInputSummary`、`SelectedSegmentContext` は定義されているが、画面遷移時に何をURL、ルーティングstate、API再取得で扱うかが統一されていない。

改善提案:
- `docs/design/navigation-context.md` を追加する。
- URLに保持するID、画面stateで持つ一時情報、APIから再取得する永続情報を分類する。
- ブラウザリロード、戻る/進む、別タブ表示、期限切れジョブの挙動を定義する。

#### 3.6 非同期ジョブの状態遷移が画面ごとに散っている
分析ジョブは `queued / running / completed / failed`、画面状態は `submitted / streaming / partial` などが定義されているが、ジョブ全体の状態機械がない。

改善提案:
- `AnalysisJobStatus` を共通定義に寄せる。
- `queued -> running -> completed|failed|cancelled` の状態遷移を明記する。
- キャンセル、タイムアウト、リトライ、重複実行、途中結果破棄の仕様を追加する。
- 結果画面で `partial` と `failed` が同時に起きるケースの表示を定義する。

#### 3.7 Fabric制約への設計が抽象的
Fabric GraphQL API経由と書かれているが、ワークスペース横断取得、サンプル値取得、行数見積もり、大量テーブル、大量カラム、クエリ負荷に対する具体的な制約設計が不足している。

改善提案:
- ワークスペース/データセット一覧はページングと遅延読み込みを前提にする。
- スキーマ取得はテーブル一覧とカラム詳細を分離し、大規模データセットで初期表示が詰まらないようにする。
- row countや件数プレビューは「厳密値」「推定値」「最終計算時刻」を区別して表示する。
- Fabric API失敗時のレート制限、再試行、バックオフ、キャッシュ方針を定義する。

#### 3.8 ノーコード体験として「自動提案の信頼性表示」が不足
自動提案は定義されているが、なぜその提案になったのか、どの程度信頼できるのか、誤提案をどう修正するかが弱い。

改善提案:
- マッピング提案、特徴量提案、セグメント候補に `confidence` と `reason` を持たせる。
- 画面上では「おすすめ理由」「確認が必要な理由」を短く表示する。
- 自動提案を一括適用する前に変更差分を表示する。

#### 3.9 アルゴリズム仕様が初期案止まり
影響度算出、パターン抽出、オートパイロットの探索方針は概要があるが、入力制約、評価指標、再現性、データリーク対策が不足している。

改善提案:
- `docs/design/analysis-logic.md` を追加する。
- 学習期間/評価期間の分割、時系列リーク防止、目的変数作成時点、特徴量窓の基準日を明記する。
- 二値/連続値ごとの評価指標を定義する。
- 乱数シード、モデルバージョン、特徴量生成バージョンを保存する。
- `lift`、`support`、`conversionDelta` の計算式を明記する。

### P2: 実装時に整理すべき事項

#### 3.10 ステータス名と型の不一致
画面ごとに `error`、`failed`、`partial`、`submitted` などが混在している。ユーザー向け状態とバックエンドジョブ状態を分離するとよい。

改善提案:
- UI状態は `loading / ready / saving / submitting / error` を基本にする。
- ドメイン状態は `draft / queued / running / completed / failed / archived` などに分ける。

#### 3.11 バリデーションIssueコードの共通化
各画面で `ValidationIssue` 相当の型が定義されているが、severity、code、message、対象フィールドの形式が揃っていない。

改善提案:
- 共通 `ValidationIssue` に `scope`、`path`、`blocking`、`suggestedAction` を追加する。
- 画面固有コードは名前空間を付ける。例: `MAPPING.MISSING_TARGET`、`ANALYSIS.INVALID_DATE_RANGE`。

#### 3.12 アクセシビリティ仕様が不足
UIの方向性はあるが、キーボード操作、スクリーンリーダー、チャート代替表現、色以外の区別が未定義である。

改善提案:
- 重要特徴量チャートやヒートマップにはテーブル代替表示を用意する。
- ステータスや重要度は色だけでなくラベルとアイコンで表現する。
- 3カラム画面でのフォーカス順、ドラッグ操作の代替ボタンを定義する。

## 4. 画面別レビュー

### 4.1 データセット選択
良い点:
- 初期導線として、接続状態、更新日時、テーブル数、推奨表示を使って業務判断を支援している。
- 一覧クエリとプレビュークエリを分ける方針は初期表示性能に適している。

不足:
- ワークスペース横断時のページング、権限エラー、レート制限が未定義。
- `recommended` 判定に必要な `primaryKeyCandidateCount` と `timestampColumnCount` が一覧クエリには含まれないため、フロントエンド計算方針と矛盾する。

改善:
- 一覧用の軽量スコア項目をAPIで返すか、推奨判定をプレビュー取得後に限定する。
- `forbidden` と `ACCESS_LIMITED` の表示・選択可否を統一する。

### 4.2 セマンティック・マッピング
良い点:
- ノーコード分析に必要な「業務意味付け」を中心に据えている。
- 保存用モデルと画面表示用モデルを分ける推奨方針は妥当。

不足:
- `customer_master` が「1テーブル以上必須」になっており、複数顧客マスタの扱いが曖昧。
- join経路、複合キー、多対多、日付粒度、タイムゾーンの仕様がない。
- サンプル値表示の権限制御が未定義。

改善:
- 顧客軸は原則1つ、複数候補は統合ルールまたは主軸選択を必須にする。
- join定義モデルを追加する。
- サンプル値はマスキング、件数上限、権限条件を定義する。

スクリーンショットレビュー:
- 左ペインで縦横両方のスクロールバーが出ており、データ探索時の操作負荷が高い。横スクロールを出さず、長いカラム名は折り返す設計にする。
- 結合条件タブでは入力項目が横一列に固定され、右端の操作が見切れやすい。ペイン幅に応じて複数行へ折り返すレスポンシブグリッドにする。
- 目的変数・特徴量タブではチェックボックスとカラム名の距離が離れ、どの項目を選択しているかが読み取りづらい。チェックボックス、名称、役割バッジを3列で近接配置する。
- 下部アクションバーが一覧の上に重なり、作業中の行を隠している。画面内の sticky 領域として扱い、内容を覆わないようにする。
- 左右ペインの最小幅指定が強く、ブラウザ幅によって画面全体の横スクロールを誘発している。右ペインは縮小可能な `minmax(0, 1fr)` 系の指定にする。

### 4.3 分析条件確認 / 実験開始
良い点:
- カスタム分析とオートパイロットを同一ジョブ基盤に流す方針は拡張しやすい。
- 実行前バリデーションを2段階にする方針は安全。

不足:
- データリーク対策、学習/評価期間の境界、対象件数しきい値が未定義。
- オートパイロットの「AI」がどの処理を担うのか、Fabric/Azure上の実行基盤が不明。
- 長時間ジョブのキャンセルやタイムアウトがない。

改善:
- 分析ロジック設計を別紙化し、評価指標、再現性、モデルバージョンを定義する。
- ジョブキャンセル、再実行、同一条件の重複実行防止を追加する。

### 4.4 結果可視化
良い点:
- ビジネスユーザーが読み解く画面として、重要特徴量、黄金パターン、セグメント候補に絞れている。
- 実行中の途中表示を想定している。

不足:
- 部分結果の信頼度、暫定結果の保存可否、再実行時の旧結果扱いが未定義。
- `support`、`lift`、`conversionDelta` の定義がなく、画面表示の解釈が揺れる。
- CSV出力の権限、件数上限、マスキングが未定義。

改善:
- 指標定義と計算式を明記する。
- `partial` の表示可能セクションと操作制限を定義する。
- 結果保存、CSV出力、セグメント引き渡しを監査対象にする。

### 4.5 セグメント作成
良い点:
- 分析結果を施策実行に変換する画面として、条件編集、件数確認、保存方法選択が整理されている。
- 件数プレビューを手動再計算にする推奨方針はAPI負荷の観点で妥当。

不足:
- 顧客フラグ付与が既存テーブル更新を伴う場合の権限、衝突、ロールバック仕様がない。
- 静的リストと動的条件セグメントの違いが実行後の再利用モデルに十分反映されていない。
- 外部連携の安全性、宛先管理、データ最小化が未定義。

改善:
- 初期版は既存顧客テーブルへの直接更新より、専用セグメントテーブルへの書き込みを優先する。
- `SegmentSaveResult` に出力種別、保存先、対象件数確定値、実行者を追加する。
- 外部連携は管理者設定が存在する宛先のみ選択可能にする。

## 5. 横断的に追加すべき設計書
優先順に以下の設計書追加を推奨する。

1. `docs/design/data-persistence.md`
   - アプリメタデータ、下書き、結果、セグメント成果物の保存責務。

2. `docs/design/graphql-contract.md`
   - 共通GraphQL型、エラー形式、ページング、JSON scalar方針。

3. `docs/design/security-and-permissions.md`
   - Azure Identity/Fabric権限、操作権限、監査ログ、PII制御。

4. `docs/design/analysis-logic.md`
   - 影響度算出、黄金パターン抽出、オートパイロット、評価指標、再現性。

5. `docs/design/job-lifecycle.md`
   - 非同期ジョブ、進捗、キャンセル、リトライ、部分結果、失敗時復旧。

6. `docs/design/navigation-context.md`
   - 画面遷移、URL設計、リロード/戻る対応、一時状態と永続状態の分離。

## 6. 結論
現行設計は、画面体験とノーコード分析の流れを説明する初期設計としては十分に有用である。ただし、このまま実装に入ると、保存先、API契約、認可、個人情報、非同期ジョブ、分析ロジックの境界が画面実装へ流出するリスクが高い。

次の `/update_design` では、まずP0項目を既存設計書へ反映し、必要な横断設計書を追加することを推奨する。特に `data-persistence`、`graphql-contract`、`security-and-permissions` の3点は実装前の前提として扱うべきである。
